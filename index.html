Como trabalho

Do Planejamento à Homologação

Minha atuação não começa nos testes — começa no entendimento do problema. Participo ativamente do refinamento, defino critérios junto ao PO e só então construo a automação que valida cada premissa levantada.

Etapa 01
Refinamento & Requisitos
Participo das cerimônias de refinamento com o PO para entender a regra de negócio antes de qualquer linha de código. Questiono fluxos de exceção, limites e impacto para o usuário.
Jira / Azure DevOps Scrum / Kanban
Etapa 02
Critérios de Aceite & BDD
Converto regras de negócio em cenários Gherkin (Given / When / Then) cobrindo fluxos de sucesso, exceção e não-funcionais. Isso elimina ambiguidade e serve como contrato entre produto, dev e QA.
Gherkin / BDD Cucumber
Etapa 03
Mocking & Isolamento
Antes de a integração estar disponível, crio stubs WireMock para simular microsserviços externos e permitir testes de integração paralelos ao desenvolvimento, acelerando o ciclo.
WireMock Microsserviços
Etapa 04
Automação de Testes
Desenvolvo suites E2E (Cypress / Playwright), testes de API (Postman / Robot Framework), automação mobile (Appium) e testes de performance (JMeter) — seguindo Page Object Model e Custom Commands para manutenibilidade.
Cypress Playwright Robot Framework Postman JMeter
Etapa 05
Validação de Dados & Código
Valido integridade entre front-end, API e banco via SQL. Analiso o código-fonte com SonarQube reportando code smells e vulnerabilidades antes que cheguem à produção.
SQL SonarQube BrowserStack
Etapa 06
Homologação & Relatório
Conduz testes regressivos E2E pós-deploy, monitoro os ambientes e elaboro relatórios com métricas de qualidade para os stakeholders — garantindo visibilidade real do estado de saúde do produto.
Regression E2E TestBook CI/CD
login_seguro.feature 
# Exemplo de critério de aceite elaborado durante refinamento com o PO

Feature: Autenticação segura de usuário
  # Regra de negócio: conta bloqueada após 3 tentativas inválidas

  Scenario: Login com credenciais válidas
    Given que o usuário está na tela de login
    When  ele informa e-mail "victor@empresa.com" e senha "Senha@123"
    And   clica em "Entrar"
    Then  deve ser redirecionado ao Dashboard
    And   o token JWT deve estar presente nos cookies

  Scenario: Bloqueio após 3 tentativas inválidas
    Given que o usuário realiza 3 tentativas com senha incorreta
    Then  a conta deve ser bloqueada por 15 minutos
    And   um e-mail de alerta deve ser enviado ao usuário

  Scenario: Tentativa de login com conta inativa  # fluxo de exceção
    Given que o usuário possui conta com status "INATIVA"
    When  tenta realizar login
    Then  deve ver a mensagem "Conta inativa. Contate o suporte."
    And   o HTTP status da API deve ser 403
